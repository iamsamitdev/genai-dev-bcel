### 1.1 Generative AI คืออะไร และทำไมนักพัฒนาต้องสนใจ

Generative AI คือ AI ที่ **สร้าง (Generate) เนื้อหาใหม่** ได้ เช่น ข้อความ โค้ด รูปภาพ เสียง ไม่ใช่แค่ "จำแนก" หรือ "ทำนาย" เหมือน AI ยุคก่อน สำหรับนักพัฒนา สิ่งที่เกี่ยวข้องที่สุดคือกลุ่ม **LLM (Large Language Model)** ที่รับ "ข้อความ" เข้าไปแล้วสร้าง "ข้อความ/โค้ด" ออกมา

```text
AI แบบเดิม (Discriminative):            Generative AI (LLM):
┌─────────────────────────────┐         ┌─────────────────────────────┐
│ รับข้อมูล → จำแนก/ทำนาย        │         │ รับ prompt → สร้างเนื้อหาใหม่   │
│ "รูปนี้คือแมวหรือหมา?"          │   ⟶    │ "เขียนฟังก์ชัน login ให้หน่อย"  │
│ ผลลัพธ์: ป้ายกำกับ (label)      │         │ ผลลัพธ์: โค้ด/ข้อความที่ยืดหยุ่น │
└─────────────────────────────┘         └─────────────────────────────┘
```

ประเด็นสำคัญคือ LLM ไม่ได้ "ค้นหาคำตอบจากฐานข้อมูล" แต่ "สร้างคำตอบทีละโทเคน" จากความน่าจะเป็นที่เรียนรู้มา นี่คือเหตุผลที่มันยืดหยุ่นมาก แต่ก็เป็นเหตุผลที่มัน **มั่วได้** (ดูหัวข้อ 2.4 เรื่อง Hallucination)

### 1.3 บทบาทของ AI ในวงจรพัฒนาซอฟต์แวร์ (SDLC)

Generative AI ไม่ได้ช่วยแค่ตอนเขียนโค้ด แต่แทรกเข้าไปได้ทุกขั้นของ SDLC:

```text
Plan ──▶ Design ──▶ Build ──▶ Test ──▶ Review ──▶ Deploy ──▶ Maintain
  │        │          │         │         │          │           │
 ร่าง     ออกแบบ     เขียน/    สร้าง     ตรวจหา    ช่วยเขียน    อธิบาย
 User    UML/ERD/  refactor   Unit      bug/       IaC/        โค้ดเก่า/
 Story   API Spec  โค้ด        Test      security   config      หา root
                                         review                  cause
```

> ⚠️ **ข้อควรระวัง:** AI ช่วยได้ทุกขั้น "ไม่ได้แปลว่าให้ปล่อยอัตโนมัติทุกขั้น" ยิ่งขั้นที่มีผลกระทบสูง (Deploy, Security) ยิ่งต้องมี **มนุษย์กำกับ (Human-in-the-loop)** เสมอ หลักคิดคือ "AI เร่งงาน มนุษย์รับผิดชอบผลลัพธ์"

### 1.5 หลักการเลือกโมเดลให้เหมาะกับงาน

การเลือกโมเดลไม่ใช่ "เอาตัวเก่งสุดเสมอ" เพราะตัวเก่งสุดมักแพงและช้าสุด ให้พิจารณา 5 แกน:

| แกนพิจารณา | คำถามที่ต้องถาม | ตัวอย่างการตัดสินใจ |
| --- | --- | --- |
| **Capability (ความสามารถ)** | งานยาก/ต้องใช้เหตุผลลึกไหม | งานออกแบบสถาปัตยกรรม เลือกตัว Flagship (Opus/GPT-5.5) |
| **Cost (ราคาต่อ Token)** | ปริมาณเรียกใช้เยอะไหม | งาน bulk เช่นแปลข้อความจำนวนมาก เลือกตัวเล็ก (Haiku/mini) |
| **Speed/Latency** | ต้องตอบเร็วแบบ realtime ไหม | Autocomplete ในโค้ด เลือกตัวเร็ว |
| **Context Window** | ต้องป้อนเอกสาร/โค้ดยาวแค่ไหน | อ่าน codebase ทั้งrepo เลือกตัว context ใหญ่ |
| **Privacy/Data** | ข้อมูลอ่อนไหว/ห้ามออกนอกองค์กรไหม | ข้อมูลลูกค้าธนาคาร พิจารณา Open-source รันเอง หรือ Enterprise plan ที่ไม่เก็บ log |

### 2.1 Tokenization - AI ไม่ได้เห็น "ตัวอักษร" แต่เห็น "โทเคน"

ก่อนที่ LLM จะประมวลผลข้อความ มันจะตัดข้อความเป็นชิ้นเล็ก ๆ เรียกว่า **Token** (ประมาณคำหรือส่วนของคำ) แล้วแปลงเป็นตัวเลข

```text
ข้อความ:  "ผมชอบเขียนโปรแกรม"
           │
           ▼  (Tokenizer ตัดเป็นชิ้น)
Tokens:   ["ผม", "ชอบ", "เขียน", "โปรแกรม"]   ← ตัวอย่างเชิง概念
           │
           ▼  (แปลงเป็น ID ตัวเลข)
IDs:      [1052, 3391, 882, 6120]
```

สิ่งที่นักพัฒนาต้องรู้จากเรื่องนี้:

- **ค่าใช้จ่ายคิดตาม Token** ไม่ใช่ตามตัวอักษร ทั้ง input และ output ยิ่ง prompt ยาว ยิ่งแพง
- **ภาษาไทย/ลาว มักกินโทเคนมากกว่าอังกฤษ** เพราะ tokenizer ส่วนใหญ่ปรับให้ภาษาอังกฤษดีที่สุด ประโยคไทยความยาวเท่ากันอาจใช้โทเคนมากกว่าอังกฤษ 2-3 เท่า
- โดยประมาณ ภาษาอังกฤษ 1 token ≈ 4 ตัวอักษร ≈ 0.75 คำ

> 📖 **ลองเล่น:** OpenAI มีเครื่องมือ Tokenizer ให้ทดลองดูว่าข้อความหนึ่งถูกตัดเป็นกี่โทเคน (platform.openai.com/tokenizer) ลองวางประโยคภาษาไทยกับอังกฤษที่ความหมายเดียวกัน แล้วเทียบจำนวนโทเคน

### 2.2 Transformer และ Attention - หัวใจที่ทำให้ LLM เข้าใจบริบท

LLM สมัยใหม่สร้างบนสถาปัตยกรรม **Transformer** ซึ่งมีกลไกสำคัญชื่อ **Attention** ที่ทำให้โมเดล "ให้ความสนใจ" กับคำที่เกี่ยวข้องกันในประโยค ไม่ว่าจะอยู่ห่างกันแค่ไหน

```text
ประโยค: "ธนาคารเปิดบัญชีให้ลูกค้า เพราะ มัน ต้องการขยายฐาน"
                                              │
                    Attention เชื่อมว่า "มัน" หมายถึง "ธนาคาร"
                    (ไม่ใช่ "ลูกค้า" หรือ "บัญชี")
```

หลักการทำงานแบบง่ายที่สุด: LLM เป็น **โมเดลทำนายโทเคนถัดไป (Next-token Prediction)** มันดูข้อความที่มีอยู่ แล้วคำนวณว่า "โทเคนถัดไปที่น่าจะเป็นมากที่สุดคืออะไร" ทำซ้ำทีละโทเคนจนจบคำตอบ

```text
Input:  "เมืองหลวงของประเทศลาวคือ"
         │
         ▼
LLM คำนวณความน่าจะเป็นของโทเคนถัดไป:
   "เวียง"  → 88%
   "หลวง"   →  5%
   "กรุง"   →  2%   ...
         │
         ▼ (เลือกตามความน่าจะเป็น)
Output: "เวียงจันทน์"
```

> ⚠️ **นัยสำคัญ:** เพราะมันเป็นการ "ทำนายทีละโทเคนจากความน่าจะเป็น" ไม่ใช่ "ค้นคำตอบจากฐานความจริง" LLM จึง **ไม่มีแนวคิดเรื่อง 'ถูก/ผิด' ในตัวเอง** มันแค่สร้างสิ่งที่ "ดูน่าจะใช่" ตามที่เรียนมา เข้าใจจุดนี้แล้ว เรื่อง Hallucination จะไม่แปลกใจอีกต่อไป

### 2.3 Context Window - "ความจำระยะสั้น" ของ AI

**Context Window** คือปริมาณโทเคนสูงสุดที่โมเดลรับได้ในครั้งเดียว (นับรวมทั้ง input + output + ประวัติแชต) เปรียบเหมือน "โต๊ะทำงาน" ที่วางเอกสารได้จำกัด

```text
┌──────────── Context Window (เช่น 200K tokens) ────────────┐
│ System Prompt │ ประวัติการสนทนา │ ไฟล์/เอกสารที่แนบ │ คำถามล่าสุด │ พื้นที่ตอบ │
└────────────────────────────────────────────────────────────┘
       ทุกอย่างต้องอยู่ในกรอบนี้ - เกินแล้วโมเดล "ลืม" ส่วนที่เก่าสุด
```

ประเด็นที่นักพัฒนาต้องรู้:

- Context Window ปี 2026 ใหญ่ขึ้นมาก (หลายแสนโทเคน บางรุ่นถึงระดับล้าน) แต่ **"ใส่ได้เยอะ ไม่ได้แปลว่าควรใส่เยอะ"**
- ยิ่งบริบทยาว โมเดลยิ่งเสี่ยง "หลงประเด็น" (Lost in the middle) - ข้อมูลกลาง ๆ มักถูกให้ความสำคัญน้อยกว่าต้นและท้าย
- นี่คือเหตุผลที่ **Context Engineering** (Module 3) สำคัญ เราต้องเลือกใส่ "เฉพาะบริบทที่จำเป็น" ไม่ใช่ยัดทุกอย่าง

### 2.4 Reasoning Model และ "Thinking" คืออะไร

โมเดลยุคใหม่ (2025-2026) มีโหมด **Reasoning / Thinking** ที่โมเดล "คิดเป็นขั้นตอนในใจ" ก่อนตอบ คล้ายการทดในกระดาษทดก่อนเขียนคำตอบสุดท้าย

```text
โมเดลรุ่นเก่า:               Reasoning Model:
Prompt → ตอบทันที            Prompt → [คิดเป็นขั้น ๆ ในใจ] → ตอบ
                                      ├─ แตกปัญหาเป็นส่วนย่อย
                                      ├─ ลองหลายแนวทาง
                                      └─ ตรวจสอบตัวเอง
เร็ว แต่พลาดโจทย์ซับซ้อน       ช้ากว่า/แพงกว่า แต่แม่นกับโจทย์ที่ต้องใช้เหตุผล
```

**เมื่อไหร่ควรใช้ Reasoning Model:** งานที่ต้องใช้ตรรกะหลายขั้น เช่น debug บั๊กซับซ้อน ออกแบบอัลกอริทึม วิเคราะห์ trade-off สถาปัตยกรรม
**เมื่อไหร่ไม่จำเป็น:** งานง่ายตรงไปตรงมา เช่น เขียน boilerplate, แปลข้อความ, format โค้ด (ใช้โหมดปกติเร็วกว่าและถูกกว่า)

### 2.5 RLHF และ Post-training - ทำไม AI ถึง "ว่านอนสอนง่าย"

โมเดลดิบที่เพิ่งเทรนจากข้อความอินเทอร์เน็ต (Pre-training) จะ "เดาโทเคนต่อ" ได้เก่งแต่ยังไม่รู้จัก "ทำตามคำสั่ง" หรือ "สุภาพ/ปลอดภัย" จึงต้องมีขั้น **Post-training** เพิ่ม โดยเฉพาะ **RLHF (Reinforcement Learning from Human Feedback)** ที่ใช้มนุษย์ให้คะแนนคำตอบ เพื่อสอนให้โมเดล:

- ทำตามคำสั่ง (Instruction Following)
- ตอบอย่างเป็นประโยชน์ ซื่อสัตย์ และไม่ก่ออันตราย (Helpful, Honest, Harmless)
- ปฏิเสธคำขอที่เป็นอันตราย

> 💡 **นัยต่อการใช้งาน:** เพราะ RLHF สอนให้โมเดล "อยากช่วย" มันจึงมีแนวโน้ม **ตอบทั้งที่ไม่รู้** (แทนที่จะบอกว่าไม่รู้) นี่เป็นอีกหนึ่งต้นตอของ Hallucination วิธีแก้คือ prompt ให้มัน "บอกได้ว่าถ้าไม่แน่ใจ" และให้บริบท/แหล่งอ้างอิงเสมอ

### 2.6 Hallucination - ทำไม AI ถึง "มั่นใจแต่ผิด"

**Hallucination** คือการที่ AI สร้างข้อมูลที่ฟังดูน่าเชื่อถือแต่ไม่จริง เช่น อ้างชื่อฟังก์ชันที่ไม่มีอยู่ อ้างเอกสาร/งานวิจัยที่ไม่มีจริง หรือมั่นชื่อ API ที่ไม่มี

สาเหตุเชิงเทคนิค:

- LLM สร้างคำตอบจาก **ความน่าจะเป็น** ไม่ใช่ฐานความจริง มันเลือกสิ่งที่ "ดูน่าจะใช่" เสมอ แม้ไม่มีข้อมูลจริงรองรับ
- มันไม่มีกลไก "รู้ว่าตัวเองไม่รู้" โดยธรรมชาติ
- ข้อมูลที่เทรนมามี **Knowledge Cutoff** (วันตัดความรู้) ถามเรื่องหลังจากนั้น มันจะเดา

| กลยุทธ์ลด Hallucination | ทำอย่างไร |
| --- | --- |
| ให้บริบท/แหล่งข้อมูลจริง | แนบเอกสาร โค้ด หรือใช้ RAG (วันที่ 3-4) |
| ขอให้อ้างอิงแหล่ง | "ตอบโดยอ้างอิงเฉพาะจากเอกสารที่ให้ ถ้าไม่มีให้บอกว่าไม่พบ" |
| อนุญาตให้ตอบ "ไม่รู้" | บอกโมเดลชัดเจนว่าเดาไม่ได้ |
| ตรวจสอบเสมอ (Verify) | โค้ดที่ AI สร้างต้อง "รันจริง/เทสต์จริง" ไม่ใช่เชื่อทันที |

> ⚠️ **กฎเหล็กของหลักสูตรนี้:** "โค้ดที่ AI เขียน ยังไม่ใช่โค้ดที่ทำงานได้ จนกว่าคุณจะรันและตรวจสอบ" ตลอด 5 วัน เราจะฝึกนิสัย **Trust but Verify** เสมอ

### 3.1 หลักการเขียน Prompt ที่ดี 4 เสา

จำง่าย ๆ ว่า Prompt ที่ดีต้อง **Clear, Specific, Role-based, Example-driven**

| เสา | หลักการ | ตัวอย่างแย่ | ตัวอย่างดี |
| --- | --- | --- | --- |
| **Clear** | ชัดเจน ไม่กำกวม | "แก้โค้ดนี้ให้หน่อย" | "แก้บั๊กที่ฟังก์ชัน `login` คืน null เมื่อ email ว่าง" |
| **Specific** | ระบุรายละเอียด/ข้อจำกัด | "เขียน API" | "เขียน REST API endpoint POST /users ด้วย Express รับ JSON {name,email} validate ด้วย zod" |
| **Role-based** | กำหนดบทบาท | (ไม่กำหนด) | "คุณคือ Senior Security Engineer ช่วยรีวิวโค้ดนี้หาช่องโหว่" |
| **Example-driven** | ให้ตัวอย่าง (Few-shot) | (ไม่มีตัวอย่าง) | "จัดรูปแบบ log ตามตัวอย่างนี้: `[2026-07-20] INFO ...`" |

**โครงสร้าง Prompt มาตรฐานที่แนะนำ:**

```text
[บทบาท]      คุณคือ ...
[บริบท]      โปรเจกต์นี้ใช้ ... โครงสร้าง ... ข้อจำกัด ...
[งาน]        ช่วย ... โดยมีขั้นตอน ...
[รูปแบบผล]   ตอบเป็น ... (โค้ด / ตาราง / JSON)
[ข้อห้าม]    ห้าม ... / ถ้าไม่แน่ใจให้ถามก่อน
```

### 3.2 เทคนิค Prompt ขั้นสูงสำหรับงาน Dev

| เทคนิค | ใช้เมื่อไหร่ | ตัวอย่าง |
| --- | --- | --- |
| **Zero-shot** | งานง่ายทั่วไป | "สรุปว่าฟังก์ชันนี้ทำอะไร" |
| **Few-shot** | อยากได้รูปแบบผลเฉพาะ | ให้ตัวอย่าง input→output 2-3 คู่ ก่อนโจทย์จริง |
| **Chain-of-Thought** | โจทย์ต้องใช้เหตุผล | "คิดทีละขั้นก่อนตอบ อธิบายเหตุผลแต่ละขั้น" |
| **Role Prompting** | อยากได้มุมมองเฉพาะทาง | "ในฐานะ DBA ช่วยดู query นี้" |
| **Constraint Prompting** | คุมขอบเขต | "ใช้เฉพาะ standard library ห้ามเพิ่ม dependency" |

**ตัวอย่างจริง - Prompt สำหรับ 5 งานหลักของ Developer:**

```text
1) Generate:  "เขียนฟังก์ชัน TypeScript ชื่อ formatCurrency(amount, currency)
              คืนสตริงรูปแบบเงิน รองรับ LAK และ THB ใส่ตัวคั่นหลักพัน พร้อม JSDoc"

2) Debug:     "โค้ดนี้ throw 'TypeError: cannot read property id of undefined'
              ที่บรรทัด 24 ช่วยหาสาเหตุและเสนอวิธีแก้ที่ปลอดภัยที่สุด"

3) Explain:   "อธิบายว่าฟังก์ชันนี้ทำงานอย่างไรทีละขั้น สำหรับ junior dev
              ชี้จุดที่อาจมีปัญหา performance ด้วย"

4) Refactor:  "รีแฟกเตอร์โค้ดนี้ให้อ่านง่ายขึ้นโดยไม่เปลี่ยนพฤติกรรม
              แยกฟังก์ชันย่อย ตั้งชื่อให้สื่อความหมาย และคง test เดิมให้ผ่าน"

5) Test:      "เขียน unit test ครอบคลุม happy path และ edge cases
              ของฟังก์ชันนี้ ด้วย Jest พร้อมคอมเมนต์อธิบายแต่ละเคส"
```

> 📌 ตัวอย่างโค้ด TypeScript/JavaScript ในเอกสารนี้ **ไม่ใส่เครื่องหมาย semicolon** ตามสไตล์โค้ดของหลักสูตร

### 3.3 Context Engineering - ยกระดับจาก "คำสั่ง" สู่ "สภาพแวดล้อม"

เมื่อเราใช้ AI Agent ทำงานยาว (เช่น สร้างฟีเจอร์ทั้งฟีเจอร์) แค่ prompt ดีไม่พอ เราต้องออกแบบ **บริบททั้งหมดที่ AI มองเห็น** ให้ถูกต้อง นี่คือ Context Engineering

```text
       สิ่งที่ประกอบเป็น "บริบท" ของ AI Agent
┌───────────────────────────────────────────────────┐
│ 1. System Prompt / Persona   (นิสัยและกติกาพื้นฐาน)  │
│ 2. ไฟล์บริบทโปรเจกต์          (AGENTS.md / CLAUDE.md)  │
│ 3. โค้ด/ไฟล์ที่เกี่ยวข้อง       (ที่ Agent อ่านเข้าไป)   │
│ 4. ประวัติการสนทนา            (สิ่งที่คุยกันมา)         │
│ 5. เครื่องมือที่ใช้ได้ (Tools)   (รันคำสั่ง/อ่านไฟล์/MCP) │
│ 6. ผลลัพธ์จากการรัน (Feedback) (error, test result)   │
└───────────────────────────────────────────────────┘
```

หลักการสำคัญของ Context Engineering:

- **ใส่เฉพาะที่จำเป็น (Relevance over Volume)** - บริบทที่ตรงประเด็นดีกว่าบริบทที่เยอะ
- **จัดโครงสร้างให้ชัด** - ใช้หัวข้อ/ป้ายกำกับ ให้ AI แยกแยะ "อะไรคือกฎ อะไรคือข้อมูล อะไรคือคำถาม"
- **สด/ถูกต้อง** - บริบทเก่าหรือผิด ทำให้ AI ทำผิดตาม
- **กันบริบทปนเปื้อน (Context Poisoning)** - ระวังข้อมูลจากภายนอกที่อาจมีคำสั่งแฝง (เรื่องนี้เชื่อมกับ Prompt Injection วันที่ 5)

### 3.4 ไฟล์บริบทมาตรฐาน: AGENTS.md และ CLAUDE.md

ปี 2026 มีธรรมเนียมให้วาง "ไฟล์กติกาของโปรเจกต์" ไว้ที่ราก repo เพื่อให้ AI Agent อ่านทุกครั้งที่เริ่มงาน ทำให้ AI รู้จักโปรเจกต์เหมือนสมาชิกใหม่ที่อ่าน onboarding doc

- **`AGENTS.md`** - มาตรฐานกลางที่หลายเครื่องมือรองรับ (Cursor, Copilot, Codex ฯลฯ)
- **`CLAUDE.md`** - ไฟล์ที่ Claude Code อ่านโดยเฉพาะ (แนวคิดเดียวกัน)

**โครงสร้างที่แนะนำของ `AGENTS.md`:**

```markdown
# AGENTS.md - กติกาสำหรับ AI Agent ในโปรเจกต์นี้

## ภาพรวมโปรเจกต์
ระบบ Internal API ของทีม X ใช้ Node.js 20 + Express + PostgreSQL

## คำสั่งที่ใช้บ่อย (Commands)
- ติดตั้ง: `npm install`
- รัน dev: `npm run dev`
- เทสต์: `npm test`
- lint: `npm run lint`

## มาตรฐานโค้ด (Conventions)
- TypeScript strict mode, ไม่ใส่ semicolon
- ตั้งชื่อไฟล์ kebab-case, ฟังก์ชัน camelCase
- ทุก endpoint ต้อง validate input ด้วย zod

## สิ่งที่ห้ามทำ (Guardrails)
- ห้ามแก้ไฟล์ใน /migrations โดยไม่ถาม
- ห้าม commit ไฟล์ .env
- ถ้าจะเพิ่ม dependency ใหม่ ให้ถามก่อนเสมอ

## โครงสร้างโฟลเดอร์
- src/routes  - API endpoints
- src/services - business logic
- src/db      - database access
```

> ✅ **ทำไมได้ผล:** ไฟล์นี้ทำให้ AI ไม่ต้อง "เดา" มาตรฐานทีม ลดการแก้ผิดสไตล์ ลดการทำสิ่งที่ห้าม และทำให้ผลลัพธ์สม่ำเสมอไม่ว่าจะเป็นสมาชิกคนไหนสั่ง เราจะได้ลงมือเขียนไฟล์นี้ใน Workshop และใช้จริงตลอดวันที่ 2-5

### 3.5 ข้อควรระวังด้านความปลอดภัยและข้อมูลตั้งแต่วันแรก

เนื่องจากผู้เรียนเป็นทีมพัฒนาของธนาคาร ต้องปลูกฝังตั้งแต่วันแรก:

- **อย่าวางข้อมูลอ่อนไหวลง AI สาธารณะ** เช่น ข้อมูลลูกค้าจริง เลขบัญชี รหัสผ่าน API key ให้ใช้ข้อมูลจำลอง (mock) ในห้องเรียนเสมอ
- **ตรวจนโยบายองค์กร** ว่าเครื่องมือ AI ตัวไหนได้รับอนุมัติให้ใช้กับข้อมูลระดับใด
- **แยกสภาพแวดล้อม** ทดลองบนโปรเจกต์ตัวอย่าง ไม่ใช่ระบบ production จริง

วันที่ 4-5 เราจะเรียก LLM ผ่าน API ให้เริ่มขอ API Key ไว้ล่วงหน้า (บางค่ายใช้เวลาอนุมัติ):

```text
[ ] Anthropic Console → console.anthropic.com  (สร้าง API Key)
[ ] OpenAI Platform   → platform.openai.com    (สร้าง API Key)
[ ] Google AI Studio  → aistudio.google.com     (สร้าง API Key)
```

> ⚠️ **สำคัญ:** API Key คือความลับ ห้าม commit ลง Git ห้ามแชร์ในแชต เก็บในไฟล์ `.env` และใส่ `.env` ใน `.gitignore` เสมอ (เราจะย้ำเรื่องนี้อีกครั้งวันที่ 4)

### 🧪 Lab 1.2 - ทดลอง Prompt เดียวกันบน 3 โมเดล แล้วเปรียบเทียบ

ใช้โจทย์เดียวกันนี้ป้อนทั้ง 3 โมเดล แล้วบันทึกผลลงตารางเปรียบเทียบ:

**โจทย์ทดสอบ (คัดลอกไปใช้ได้เลย):**

```text
คุณคือ Senior TypeScript Developer
ช่วยเขียนฟังก์ชัน isValidLaoPhone(phone: string): boolean
ตรวจสอบว่าเป็นเบอร์โทรลาวที่ถูกต้องหรือไม่ (ขึ้นต้น 020 ตามด้วยตัวเลข 8 หลัก)
- เขียนแบบไม่ใส่ semicolon
- ใส่ JSDoc อธิบาย
- เขียน unit test 4 เคส (ถูก 1, ผิด 3) ด้วย Vitest
- อธิบายสั้น ๆ ว่าทำไมออกแบบ regex แบบนี้
```

**ตารางบันทึกผลเปรียบเทียบ:**

| เกณฑ์ | Claude | ChatGPT | Gemini |
| --- | --- | --- | --- |
| โค้ดถูกต้อง รันได้ทันที | | | |
| ทำตามกติกา (ไม่มี semicolon) | | | |
| คุณภาพ test ที่เขียน | | | |
| ความชัดของคำอธิบาย | | | |
| ความเร็วในการตอบ | | | |

> 💡 **สิ่งที่ต้องสังเกต:** แต่ละโมเดลมี "บุคลิก" ต่างกัน บางตัวละเอียด บางตัวกระชับ บางตัวทำตามข้อจำกัดเป๊ะกว่า จุดประสงค์ไม่ใช่หา "ตัวชนะ" แต่คือเข้าใจว่า "งานแบบไหนควรใช้ตัวไหน"

### 🧪 Lab 1.3 - ฝึก Prompt 5 งานหลักของ Dev

เลือกโค้ดสั้น ๆ ที่ผู้เรียนคุ้นเคย (หรือใช้ snippet ที่วิทยากรแจก) แล้วลองสั่ง AI ทำครบทั้ง 5 งาน: Generate, Debug, Explain, Refactor, Test โดยใช้โครงสร้าง Prompt จากหัวข้อ 3.1

บันทึกไว้ว่า Prompt แบบไหนได้ผลดี Prompt แบบไหนต้องแก้ - นี่คือจุดเริ่มต้นของ **Prompt Library ส่วนตัว** ที่เราจะสะสมต่อทั้งสัปดาห์

### 🛠️ Workshop หลัก - เขียน AGENTS.md ฉบับแรกของคุณ

**โจทย์:** สมมติผู้เรียนกำลังเริ่มโปรเจกต์ Internal Tool เล็ก ๆ (เลือก stack ที่ถนัด เช่น Node.js + Express หรือ Python + FastAPI) ให้เขียนไฟล์ `AGENTS.md` ที่ราก repo ตามโครงสร้างหัวข้อ 3.4 ให้ครบ 6 ส่วน:

```text
[ ] 1. ภาพรวมโปรเจกต์ (โปรเจกต์ทำอะไร ใช้ stack อะไร)
[ ] 2. คำสั่งที่ใช้บ่อย (install / dev / test / lint)
[ ] 3. มาตรฐานโค้ด (naming, style, ภาษา)
[ ] 4. สิ่งที่ห้ามทำ (guardrails อย่างน้อย 3 ข้อ)
[ ] 5. โครงสร้างโฟลเดอร์
[ ] 6. หมายเหตุพิเศษของทีม (เช่น เรื่องความปลอดภัยของธนาคาร)
```

**ทดสอบว่าไฟล์ได้ผลจริง:** เปิดแชตใหม่กับ AI วางเนื้อหา `AGENTS.md` เข้าไป แล้วสั่งงานเช่น "ช่วยสร้าง endpoint สำหรับดึงรายการผู้ใช้" สังเกตว่า AI ทำตามมาตรฐานที่เรากำหนดหรือไม่ (เช่น validate input, ไม่ใส่ semicolon, ตั้งชื่อตาม convention)

### 🧩 ความท้าทายเสริม (ถ้าทำเสร็จก่อนเวลา)

- ลองเพิ่มส่วน "ตัวอย่างโค้ดที่ดี (Golden Example)" ลงใน `AGENTS.md` แล้วดูว่า AI เลียนแบบสไตล์ได้ดีขึ้นไหม
- ทดลองถาม AI คำถามที่มัน "ควรตอบว่าไม่รู้" (เช่น เรื่องหลัง knowledge cutoff) สังเกตว่ามัน hallucinate หรือยอมรับว่าไม่รู้
- เปรียบเทียบจำนวนโทเคนของ prompt ภาษาไทยกับอังกฤษด้วย OpenAI Tokenizer
